The big idea

Three moves, repeated

Quick Sort is a divide-and-conquer algorithm. It never sorts the whole list at once — it keeps placing one element perfectly, then trusting recursion for the rest.

i.

Choose a pivot

Pick one element as the reference point. Everything else will be measured against it. Here the pivot defaults to the last element of the current range.

ii.

Partition

Walk through the range and shuffle values so that everything smaller than the pivot ends up on its left, everything larger on its right. The pivot lands in its final, sorted home.

iii.

Conquer each side

The two sides are now independent, smaller sorting problems. Apply the exact same recipe to each, until ranges shrink to a single element — already sorted.

Inside the partition (Lomuto scheme)

The visualizer uses the Lomuto partition because it maps cleanly to what you see on screen. Two markers do all the work:

  • j — the scanner. It visits every element in the range except the pivot, asking a single question: “are you smaller than or equal to the pivot?”
  • i — the boundary. It marks the edge of the “smaller” region. Whenever j finds a small value, i steps forward and the two values swap, growing the smaller region by one.

When the scan finishes, every value left of the boundary is smaller than the pivot and everything right is larger. One last swap drops the pivot exactly between them — and that position is now permanently correct. The range splits there and recursion takes over.

What it costs

Fast on average, fragile at worst

Quick Sort is usually the fastest general-purpose sort in practice — but a careless pivot on already-sorted data can drag it down to quadratic time.

When to use Quick Sort

Use Quick Sort when you want fast in-memory sorting, stability does not matter, and you can choose a pivot strategy that avoids consistently unbalanced partitions.

  • General-purpose sorting for arrays held in memory.
  • Cases where low extra memory use is important.
  • Data where average-case speed matters more than stable ordering.
CaseTimeWhy
BestO(n log n)Each pivot splits its range roughly in half, giving a shallow, balanced tree.
AverageO(n log n)Random data tends to split reasonably well, so this is the everyday cost.
WorstO(n²)Pivots that are always the smallest or largest (e.g. last-element pivot on sorted input) create a lopsided tree of depth n.
SpaceO(log n)In-place sorting; the only extra memory is the recursion stack, log-deep when balanced.
